# 提炼与收录标准

case 拆解表格里 ✅ 的判断依据。四条按优先级排列,前两条一票否决:

## 1. 复用性(否决项)

脱离原站语境是否还成立?
- ✅ Lando 的 hover 双图切换:任何画廊都能用
- ❌ Bruno Simon 的"开车导航":离开他的世界观就不成立 → 只留在 case 里

## 2. 可复现性(否决项)

能否做出**有教学价值的简化版**零依赖单文件 demo?不要求复现原效果的完成度,要求保留核心机制。
- ✅ 滚动驱动 shader:可以做最小版(一个 uniform)
- ✅ WebGL 图片 displacement 过渡:原站管线复杂,但"一张噪声图驱动 mix()"的最小版机制完整
- ❌ Igloo 的晶体生长算法:简化后机制本身就没了 → 降级为 case 里的"方法论笔记"

> 2026-07-16 口径调整:原标准是"能否复现原效果",实践发现把大量高频效果挡在门外(简化版明明成立)。改为以"核心机制是否保得住"判断。

## 3. 成本收益比(排序项)

实现成本低、感知强的优先做。双图切换(两张图 + opacity)先于 GPU 粒子。

## 4. 出现频率(排序项)

多个 case 反复出现 = 行业通用语言,优先级高于单站私货。
- 例:逐字/逐行拆分入场在 Lando、Hashgraph、Robin 三站实测撞车 → 最高优先级

## 5. 分类覆盖度(排序项)

taxonomy 八类中还是空的分类(如 cursor/),同等条件下优先补;库的价值在"用的时候有得选",偏科会让空分类永远空着。

## 不抽成 pattern 的去向

- **方法论笔记**:留在 case 文档里(如 Messenger 的 5.7MB 体积预算、Igloo 的 scroll-datatexture 烘焙、Bruno 的阴影烘焙)
- **工具依赖型**:依赖特定工具的(Rive 动画)记录在 tech-radar,不做 demo

> 本标准可议,改这份文档即可调整后续收录口径。
